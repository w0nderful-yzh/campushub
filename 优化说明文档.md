# Campushub 存储过程和触发器优化说明

## 优化概述

本次优化主要针对现有存储过程和触发器的以下方面进行改进：
- 事务安全性
- 并发处理
- 代码简化
- 性能优化
- 返回格式统一

---

## 一、存储过程优化

### 1. sp_get_post_full_detail (获取帖子详情)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 结果集 | 返回2个结果集 | 合并为单个结果集，评论以JSON格式返回 |
| 事务 | 无 | 使用READ ONLY事务 |
| 浏览量更新 | 无条件更新 | 验证帖子有效性后更新 |

**优化原因**：
- 多结果集在某些ORM框架中可能导致兼容性问题
- READ ONLY事务提高查询性能
- 避免对已删除帖子更新浏览量

---

### 2. sp_get_user_dashboard (用户仪表盘)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 结果集 | 返回3个结果集 | 合并为单个结果集，使用JSON格式 |
| 查询优化 | 多次查询 | 使用子查询减少数据库往返 |

**优化原因**：
- 减少客户端与数据库之间的往返次数
- JSON格式更灵活，便于前端处理
- 统一返回格式

---

### 3. sp_handle_report (处理举报)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 事务 | 无 | 使用显式事务 |
| 查询优化 | 多次单独查询 | 合并验证查询 |
| 并发控制 | 无 | 使用FOR UPDATE锁 |

**优化原因**：
- 事务确保数据一致性
- FOR UPDATE防止并发处理同一举报
- 减少数据库往返

---

### 4. sp_publish_post (发布帖子)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 事务 | 无 | 使用显式事务 |
| 验证优化 | 两次单独查询 | 先合并验证，失败时再精确报错 |

**优化原因**：
- 事务确保发布操作的原子性
- 合并验证减少正常情况下的查询次数
- 保持精确的错误提示

---

### 5. sp_signup_activity (活动报名)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 返回方式 | OUT参数 + 结果集 | 统一结果集返回 |
| 并发控制 | 无 | 使用FOR UPDATE锁 |
| 事务 | 无 | 使用显式事务 |

**优化原因**：
- 移除OUT参数，统一返回格式
- FOR UPDATE防止并发超卖（名额竞争）
- 事务确保操作原子性

---

### 6. sp_toggle_post_like (切换点赞)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 返回方式 | OUT参数 + 结果集 | 统一结果集返回 |
| 并发控制 | 无 | 使用FOR UPDATE锁 |
| 事务 | 无 | 使用显式事务 |

**优化原因**：
- 移除OUT参数，统一返回格式
- FOR UPDATE防止并发重复点赞
- 事务确保计数准确性

## 二、触发器优化

### 1. trg_activity_signup_after_insert (活动报名触发器)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 冗余检查 | 包含名额验证 | 移除，由存储过程负责 |

**优化原因**：
- 存储过程已做名额检查和锁控制
- 触发器中重复检查增加不必要的开销
- 职责分离：存储过程负责业务逻辑，触发器负责数据同步

---

### 2. trg_comment_after_insert (评论插入触发器)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 变量使用 | 使用v_notice_text变量 | 直接在INSERT中构建 |
| 代码简化 | 冗余变量声明 | 简化逻辑 |

**优化原因**：
- 减少不必要的变量声明和赋值
- 代码更简洁易读

---

### 3. trg_comment_after_update (评论更新触发器)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 条件判断 | 多个IF ELSEIF分支 | 使用变量简化判断逻辑 |
| 可读性 | 复杂条件嵌套 | 清晰的状态变化判断 |

**优化原因**：
- 使用v_old_active和v_new_active变量简化条件
- 逻辑更清晰，易于维护

---

### 4. trg_favorite_after_insert/delete (收藏触发器)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 通知内容 | 包含帖子标题 | 简化为固定文案 |
| 删除条件 | 包含时间限制 | 移除时间限制 |

**优化原因**：
- 简化通知内容，减少字符串拼接开销
- 移除时间限制，确保通知一致性删除

---

### 5. trg_like_after_insert/delete (点赞触发器)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 通知内容 | 包含帖子标题 | 简化为固定文案 |
| 删除条件 | 包含时间限制 | 移除时间限制 |

**优化原因**：
- 与收藏触发器保持一致
- 简化逻辑，提高性能

---

### 6. trg_vote_record_after_insert/delete (投票记录触发器)

| 优化项 | 原版 | 优化版 |
|--------|------|--------|
| 计数保护 | 无 | 使用GREATEST确保不为负 |

**优化原因**：
- 防止数据异常导致计数变为负数
- 与like/comment等触发器保持一致

---

## 三、整体优化原则

### 1. 事务使用
- 所有写操作使用显式事务
- 查询操作使用READ ONLY事务
- 使用FOR UPDATE进行悲观锁控制

### 2. 返回格式统一
- 移除OUT参数，统一使用结果集返回
- 使用JSON格式返回复杂数据结构
- 保持返回格式一致性

### 3. 职责分离
- 存储过程：业务逻辑、验证、事务控制
- 触发器：数据同步、计数维护、通知生成

### 4. 并发控制
- 关键操作使用SELECT FOR UPDATE
- 使用唯一约束防止重复数据
- 计数字段使用GREATEST保护

### 5. 性能优化
- 合并多次查询为单次查询
- 减少变量声明和赋值
- 简化字符串拼接

## 四、部署建议

   ```bash
   docker exec dev-mysql8 mysqldump -u root -p123456 campushub > backup_before_optimize.sql
   ```

2. **执行优化脚本**
   ```bash
   docker exec -i dev-mysql8 mysql -u root -p123456 campushub < optimized_sp_triggers.sql
   ```

3. **验证结果**
   ```sql
   SHOW PROCEDURE STATUS WHERE Db = 'campushub';
   SHOW TRIGGERS FROM campushub;
   ```
